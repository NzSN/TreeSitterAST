## Implementation Details

### Parent Map Detection
- `computeParentMap` scans all rule names in the transformed grammar
- For each rule name containing an underscore, splits into prefix (parent) and suffix
- If the parent rule exists and is an empty `Choice []` (i.e., a top‑level CHOICE that has been emptied), adds mapping `alternative -> parent`
- This mapping is used in `build` to determine the correct base class

### Inheritance Logic
- In `build`, after determining the default base class (`SyntaticLeaf` or `SyntaticInterior`), checks whether the rule name appears in the parent map
- If yes, uses the parent CHOICE class (computed via `node_type_ident`) as the base class
- All other generation (fields, constructors, evaluate methods) remains unchanged

### Ordering Fix
- Rule names are sorted using `compareRules` comparator that ensures parent CHOICE classes come before their alternatives
- Comparator logic: if rule A has parent B, then B should come before A in sorted order
- Uses `L.foldl'` (left fold) to process rules in sorted order, generating TypeScript classes in correct definition order
